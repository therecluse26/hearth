## Competitor Reports

### 1. Keycloak

**Latest version:** 26.6.1 (April 15, 2026, latest stable) + nightly build active
**Target audience:** Self-hosters, enterprise on-prem
**Tier:** OSS (free) + Red Hat commercial support

**Findings (2026):**

#### 26.6.0 — JWT Authorization Grant (RFC 7523) now SUPPORTED — **HIGH**
Keycloak clients can obtain tokens using externally-signed JWTs (from a trusted OIDC IdP, Kubernetes service account, or SPIFFE JWT SVID), eliminating shared secrets entirely in M2M flows. Was preview in 26.5; promoted to supported in 26.6.
- Source: https://www.keycloak.org/docs/latest/release_notes/index.html#jwt-authorization-grant-supported
- **Impact if Hearth slips:** Machine-to-machine auth without shared secrets is a growing enterprise requirement (k8s workload identity). Hearth's client_credentials flow requires a secret; no workload-identity alternative exists yet.

#### 26.6.0 — Federated Client Authentication now SUPPORTED — **HIGH**
Clients authenticate using SPIFFE JWT SVIDs, Kubernetes service account tokens, or external OIDC IdP tokens. SPIFFE variant stays preview (underlying spec still draft). Removes need for per-client secrets.
- Source: https://www.keycloak.org/docs/latest/release_notes/index.html#federated-client-authentication-supported

#### 26.6.0 — Organization Groups in OIDC Tokens and SAML Assertions — **HIGH**
Organizations now support isolated group hierarchies (teams/departments per org, no naming conflicts across realms). IdP mappers auto-assign federated users to org groups via external claims. Group membership auto-included in OIDC tokens and SAML assertions when org context is requested.
- Source: https://www.keycloak.org/docs/latest/release_notes/index.html#organization-groups
- **Impact if Hearth slips:** Hearth has Organizations but no per-org group hierarchies. Enterprise buyers expect group-to-token mapping for RBAC delegation.

#### 26.6.0 — Step-up Authentication for SAML (preview) — **MEDIUM**
Adaptive/step-up authentication now available in SAML flows. Extends existing step-up (OIDC) coverage to SAML.

#### 26.6.0 — Zero-downtime Patch Releases — SUPPORTED — **LOW (ops)**
Patch upgrades (26.x.y) no longer require downtime. Zero-downtime now guaranteed for patch-level releases. Breaking for operator runbooks that assumed downtime.

#### Passkeys integrated into default login forms (26.3.0) — **HIGH**
Keycloak 26.3.0 added a single-click toggle (**Authentication → Policies → WebAuthn Passwordless Policy → Enable Passkeys**) that integrates passkeys support — conditional UI and modal UI — directly into the default login forms. Previously, passkeys required custom configuration. Feature is still labeled "preview."

- Source: https://www.keycloak.org/docs/26.6.1/server_admin/#passkeys_server_administration_guide
- **Impact if Hearth slips:** Keycloak's one-click passkeys lowers the setup bar for self-hosted IdPs. Hearth's WebAuthn implementation (Phase 1 Step 24) needs an equivalently frictionless operator UX to avoid a positioning deficit vs Keycloak in the self-host market.

#### OID4VCI (OpenID for Verifiable Credential Issuance) — nightly/experimental — **MEDIUM**
Keycloak began OID4VCI experimental support in 26.0.0 (pre-authorized code flow, JWT-VC / SD-JWT-VC / VCDM formats). Active development continues in nightly: a new client policy to require a credential offer was recently merged. OAuth SIG group is active here.

- Source: https://github.com/keycloak/keycloak/releases/tag/nightly (nightly build notes)
- Source: https://www.keycloak.org/docs/26.6.1/server_admin/#_oid4vci
- **Impact if Hearth slips:** OID4VCI is not yet a ratified spec; Keycloak treating it as experimental is appropriate. Hearth does not need to act immediately, but should monitor — Keycloak's involvement in the OAuth SIG accelerates the spec's maturation.

#### Persistent user sessions (GA preview since 25.0.0) — **LOW**
Online user sessions now survive cluster restarts/upgrades when the `persistent-user-sessions` feature flag is enabled. Operational durability improvement, not a new auth flow.

**Severity summary:** HIGH (passkeys UX bar), MEDIUM (OID4VCI experimental), LOW (session persistence)

| Field | Value |
|-------|-------|
| RFC 9635 (GNAP Core) | **Published and ratified (Proposed Standard)** |
| GNAP WG Status | **Concluded** — formally closed |
| Source | https://datatracker.ietf.org/wg/gnap/about/ ✅ retrieved |

**Assessment:** GNAP WG is formally **concluded**. The closing note from the responsible AD: "With the last of planned work items just approved the WG is being closed." RFC 9635 is the ratified GNAP Core Protocol. The GNAP Resource Servers spec is still in draft, but the WG is closed — further work would require a recharter.

**Stage change confirmed this cycle:** WG concluded. This moves GNAP from "active working group" to "stable ratified RFC." Hearth should monitor for ecosystem adoption signals. No implementation urgency — none of the six tracked competitors ship GNAP support. **Investment is justifiable now that the spec is stable, but market pull is still absent.**
